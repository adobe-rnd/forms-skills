# Claude Suggestions

## Suggested CLAUDE.md Additions

| Section | Suggested addition | Why |
| --- | --- | --- |
| `## AEM Forms Plugin` | When working with AEM Forms plugins/skills, always read skill files from the installed plugin directory such as `.claude/plugins/`, not from the user's source code repository. Never assume skill file paths; verify the installed plugin path first. | Multiple sessions read skills from the wrong path, which caused repeated corrections and wasted time. |
| `## AEM Forms Plugin` → `### Workspace Setup Checklist` | For AEM EDS form workflows, always follow this order: 1. Verify plugin is loaded with `/reload-plugins` if needed. 2. Verify workspace credentials are filled in. 3. Verify connectivity before form generation. Do not skip setup verification. | Several sessions failed because of unloaded plugins, missing credentials, or skipped connectivity checks. |
| `## AEM Forms Plugin` → `### Deployment Checklist` | Before deploying to AEM, always check: 1. Bearer tokens are valid. 2. Files are placed in the correct directory paths. 3. `package-lock.json` is not modified and no ESLint violations are introduced. Run `npm run lint` before committing. | Repeated deployment failures came from expired tokens, wrong file placement, lockfile changes, and lint issues. |
| `## Code Style & Conventions` | When implementing form UI components, prefer component-based approaches over direct DOM manipulation. Ask the user before choosing between alternatives such as dropdown vs autocomplete or different toast patterns. | Some sessions required rework because the implementation approach did not match the intended UX pattern. |
| `## AEM Forms Plugin` → `### Form Validation` | Use `constraintMessages` carefully in `form.json` and validate the schema before writing. For validation rules, test with a small subset first before applying many rules at once. | Incremental validation would have caught `constraintMessages` and bulk rule failures earlier. |

## New Ways to Use Claude Code

| Pattern | Summary | Why it matters | Prompt to try |
| --- | --- | --- | --- |
| Encode your plugin path knowledge once | Add explicit plugin directory paths to `CLAUDE.md` so Claude always reads skills from the correct location. | Wrong plugin path resolution appeared across multiple sessions and cascaded into broader workflow failures. | `Read the AEM forms plugin skills ONLY from .claude/plugins/aem-forms/ directory. List all available skills and their file paths before starting any form workflow.` |
| Break long sessions into phased checkpoints | Use explicit phase gates instead of running a 6 to 10 step workflow in one pass. | Smaller checkpoints reduce compounding errors and make corrections cheaper. | `We're doing Phase 3 of 6: Generate the form.json with visibility rules. Before writing any files, show me the plan and list of rules. After generating, run validation and show me the results. Do NOT proceed to Phase 4 until I confirm.` |
| Front-load connectivity and credential checks | Verify AEM tokens, connectivity, and write paths before starting the main workflow. | Several sessions failed late because auth or connectivity problems were discovered only after work had already been done. | `Before we start any form work today, verify my AEM setup: 1) Check if the bearer token in .env is still valid by making a test API call, 2) Confirm the target AEM instance is reachable, 3) Verify file write paths are correct. Report any issues before proceeding.` |
